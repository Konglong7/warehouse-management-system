## 🎤 常见答辩问题及回答

### 技术类问题

#### Q1：为什么选择 SpringBoot 而不是 SSM 框架？
**回答：**
```
SpringBoot 是 Spring 生态的集大成者，它简化了 Spring 应用的配置和部署：
1. 约定优于配置，减少了大量的 XML 配置
2. 内置 Tomcat，无需单独部署
3. 提供了大量的 starter，依赖管理更方便
4. 开发效率更高，更符合现代企业级开发的需求
```

#### Q2：为什么选择 Vue 3 而不是 Vue 2 或 React？
**回答：**
```
选择 Vue 3 的原因：
1. Vue 3 是 Vue 的最新版本，性能更好，体积更小
2. Composition API 使代码组织更灵活，逻辑复用更方便
3. 对 TypeScript 的支持更好
4. Vue 的学习曲线相对平缓，文档完善
5. 国内社区活跃，生态丰富
```

#### Q3：前后端是如何进行数据交互的？
**回答：**
```
前后端通过 RESTful API 进行数据交互：
1. 前端使用 Axios 发送 HTTP 请求
2. 后端使用 SpringBoot 提供 RESTful 接口
3. 数据格式采用 JSON
4. 前端配置了代理，将 /api 开头的请求转发到后端的 8080 端口
5. 后端返回统一的响应格式，包含状态码、消息和数据
```

#### Q4：数据库是如何设计的？
**回答：**
```
数据库设计遵循三范式，主要包含 6 张核心表：
1. sys_user：用户表，存储用户信息和角色
2. wms_supplier：供应商表，存储供应商信息
3. wms_material：物料表，存储物料信息和当前库存
4. wms_location：仓位表，存储仓位信息和状态
5. wms_inbound：入库记录表，记录入库操作
6. wms_outbound：出库记录表，记录出库操作

表之间通过外键关联，保证数据完整性。同时创建了索引提升查询性能，
创建了视图简化复杂查询。
```

#### Q5：如何保证库存数据的准确性？
**回答：**
```
系统通过以下方式保证库存数据的准确性：
1. 入库时自动增加物料的 current_stock 字段
2. 出库时自动减少物料的 current_stock 字段
3. 出库前会校验库存是否充足，不足则提示错误
4. 使用数据库事务保证操作的原子性
5. 物料表的 current_stock 字段设置了默认值 0
```

#### Q6：系统的安全性如何保证？
**回答：**
```
系统的安全性措施包括：
1. 用户登录验证，未登录无法访问系统
2. 角色权限控制，管理员和普通用户权限不同
3. 密码存储（实际项目中应该加密，可以说明这是改进方向）
4. SQL 注入防护：使用 MyBatis-Plus 的参数化查询
5. XSS 防护：前端对用户输入进行校验和转义
```

### 功能类问题

#### Q7：系统的核心功能是什么？
**回答：**
```
系统的核心功能是出入库管理和库存管理：
1. 入库管理：记录物料入库信息，自动更新库存
2. 出库管理：记录物料出库信息，自动扣减库存
3. 库存管理：实时查看物料库存，低库存预警
4. 数据统计：通过图表展示出入库趋势，辅助决策
```

#### Q8：库存预警是如何实现的？
**回答：**
```
库存预警的实现逻辑：
1. 物料表中有 warning_stock（预警库存）和 current_stock（当前库存）字段
2. 前端查询物料列表时，比较这两个字段
3. 当 current_stock <= warning_stock 时，在界面上标红显示
4. 管理者可以在物料管理中设置每个物料的预警值
5. 这样可以及时发现库存不足的物料，提醒管理者补货
```

#### Q9：如果同一个物料存放在多个仓位，系统如何处理？
**回答：**
```
系统的设计思路是：
1. 物料表记录物料的总库存（current_stock）
2. 入库和出库记录表记录具体的仓位信息（location_id）
3. 通过入库和出库记录，可以追溯每个仓位的物料存放情况
4. 如果需要查询某个物料在各个仓位的分布，可以通过统计入库和出库记录来实现
5. 这是一个可以改进的地方，未来可以增加"仓位库存表"来记录每个仓位的实时库存
```

#### Q10：系统支持哪些查询功能？
**回答：**
```
系统支持多种查询功能：
1. 物料查询：按物料编码、名称、分类查询
2. 供应商查询：按供应商编码、名称查询
3. 入库记录查询：按入库单号、物料、时间范围查询
4. 出库记录查询：按出库单号、物料、时间范围查询
5. 库存查询：查看所有物料的实时库存
6. 预警查询：筛选出库存不足的物料
```

#### Q11：系统的设计思路是什么？
**回答：**
```
系统的设计思路：
1. 需求分析：分析仓库管理的业务流程和核心需求
2. 功能设计：划分功能模块，设计功能结构图
3. 数据库设计：设计数据表结构，建立表关系
4. 接口设计：设计 RESTful API 接口
5. 界面设计：设计用户界面，注重用户体验

7. 测试优化：功能测试、性能优化
```

#### Q12：如果让你改进这个系统，你会怎么做？
**回答：**
```
系统的改进方向：
1. 安全性增强：
   - 密码加密存储（使用 BCrypt）
   - 添加 JWT 身份验证
   - 添加操作日志记录

2. 功能扩展：
   - 增加仓位库存表，精确记录每个仓位的库存
   - 增加盘点功能，定期核对库存
   - 增加报表导出功能（Excel、PDF）
   - 增加物料批次管理（生产日期、有效期）
   - 增加移动端支持（响应式设计或小程序）

3. 性能优化：
   - 添加 Redis 缓存，提升查询速度
   - 对大数据量查询进行分页优化
   - 使用消息队列处理异步任务

4. 用户体验：
   - 增加操作引导和帮助文档
   - 优化界面交互，提升易用性
   - 增加数据导入导出功能
```

#### Q13：系统的难点是什么？如何解决的？
**回答：**
```
开发过程中的主要难点：

1. 库存自动更新
   难点：入库和出库操作需要同时更新物料库存
   解决：使用数据库事务保证操作的原子性，在 Service 层添加 @Transactional 注解

2. 前后端数据交互
   难点：前端如何调用后端接口，如何处理跨域问题
   解决：使用 Axios 发送请求，在 Vite 中配置代理解决跨域

3. 数据可视化
   难点：如何将数据以图表形式展示
   解决：使用 ECharts 图表库，封装图表组件，传入数据即可渲染

4. 权限控制
   难点：如何区分管理员和普通用户的权限
   解决：在用户表中添加 role 字段，前端根据角色显示不同的菜单和功能
```
