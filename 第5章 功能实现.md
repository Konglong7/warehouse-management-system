# 第5章 功能实现

在完成系统设计之后，就进入了具体的功能实现阶段。这一章会详细介绍系统各个功能模块的实现过程，包括用户登录、物料管理、入库出库、数据可视化等核心功能。

## 5.1 用户登录功能实现

用户登录是系统的入口功能，用户必须通过身份验证才能进入系统使用其他功能。

### 5.1.1 登录界面设计

登录界面采用左右分栏的布局方式。左侧展示系统的品牌信息和特色功能介绍，右侧是登录表单。整个页面使用了渐变色背景，配合动画效果，看起来比较现代化。

登录表单包含用户名和密码两个输入框。输入框使用了圆角设计，并且在获得焦点时会有边框颜色变化和阴影效果，提升了交互体验。表单下方显示了默认的测试账号信息，方便测试使用。

### 5.1.2 登录功能实现

登录功能的实现流程比较简单。用户在前端输入用户名和密码后，点击登录按钮，前端会向后端发送登录请求。

后端接收到请求后，在UserService中查询数据库，验证用户名和密码是否正确。如果验证通过，生成一个token返回给前端。这个token的生成方式比较简单，就是"token_"加上用户ID。考虑到这是毕业设计项目，没有实现特别复杂的JWT token机制。

前端收到登录成功的响应后，会将用户信息和token保存到localStorage中，然后跳转到系统首页。同时使用Pinia状态管理库保存用户状态，方便在其他页面使用。

在路由配置中设置了路由守卫，每次路由跳转时都会检查localStorage中是否存在token。如果用户没有登录就访问系统页面，会自动跳转到登录页。如果已经登录的用户访问登录页，会自动跳转到首页。

## 5.2 系统首页功能实现

系统首页是用户登录后看到的第一个页面，展示了系统的核心数据统计和图表分析。

### 5.2.1 数据统计卡片

首页顶部有四个统计卡片，分别显示总库存、今日入库、今日出库和预警物料数量。每个卡片都使用了不同的渐变色背景，配合图标和数字，一眼就能看出各项数据。

这些统计数据是通过调用后端的统计接口获取的。后端StatsController提供了dashboard接口，这个接口会查询物料表、入库表、出库表，计算出各项统计数据。总库存是所有物料的当前库存之和，预警物料数量是通过查询当前库存小于等于预警库存的物料数量得到的。

卡片还加了一些交互效果，鼠标悬停时会有轻微的上浮动画和阴影变化，让页面看起来更有活力。

### 5.2.2 数据可视化图表

首页下方展示了两个图表：库存统计柱状图和物料分类占比饼图。

库存统计柱状图使用ECharts实现，展示了库存量最多的前10种物料。图表使用了渐变色填充，柱子顶部是圆角设计，看起来比较美观。鼠标悬停在柱子上时会显示详细的物料名称和库存数量。

物料分类占比饼图展示了不同分类物料的库存占比。饼图使用了环形设计，中间是空心的，看起来更加现代。每个扇区都有不同的颜色，并且显示了分类名称和百分比。

图表数据是通过调用后端的inventory接口获取的，这个接口返回所有物料的信息。前端拿到数据后，对于柱状图，取前10条数据；对于饼图，需要先按分类进行汇总，计算每个分类的总库存。

在实现过程中遇到了一个问题，就是窗口大小改变时图表不会自动调整。后来在代码里加了一个窗口resize事件监听，当窗口大小改变时调用图表的resize方法，这样图表就能自适应窗口大小了。

## 5.3 物料管理功能实现

物料管理是系统的基础功能，管理着仓库中所有物料的信息。

### 5.3.1 物料列表展示

物料管理页面使用表格展示所有物料信息。表格包含物料编码、物料名称、分类、单位、当前库存、预警库存等列。表格使用了斑马纹样式，奇偶行颜色不同，方便查看。

页面顶部有一个搜索框，可以按物料名称或编码进行搜索。搜索功能是在前端实现的，通过Vue的计算属性过滤数据列表。输入搜索关键词后，表格会实时显示匹配的物料。

表格右侧有操作列，每行都有编辑和删除按钮。点击编辑按钮会弹出对话框，显示物料的详细信息，可以修改后保存。点击删除按钮会弹出确认对话框，确认后调用删除接口。

### 5.3.2 物料新增和编辑

点击页面右上角的"新增物料"按钮，会弹出一个对话框。对话框中包含物料的各个字段，包括物料编码、物料名称、分类、单位、预警库存、当前库存。

新增物料时，需要填写所有必填字段。物料编码必须是唯一的，如果输入了已存在的编码，后端会返回错误提示。预警库存和当前库存使用数字输入框，可以通过点击上下箭头或直接输入数字来设置。

编辑物料的对话框和新增是同一个，只是会预先填充物料的现有数据。修改后点击保存，会调用后端的更新接口。

后端的实现比较简单，使用MyBatis Plus提供的insert和updateById方法就能完成。物料编码的唯一性是通过数据库的唯一索引来保证的，如果插入重复的编码，数据库会抛出异常。

### 5.3.3 库存预警功能

库存预警是物料管理的一个重要功能。每个物料都可以设置一个预警库存值，当物料的当前库存低于或等于这个值时，就认为该物料需要补货。

在MaterialService中实现了getWarningList方法，这个方法使用MyBatis Plus的QueryWrapper构建查询条件，查询当前库存小于等于预警库存的物料。查询条件使用了apply方法，直接写SQL条件"current_stock <= warning_stock"。

系统首页的预警物料数量统计就是调用这个方法得到的。在实际使用中，管理员可以通过这个统计及时了解哪些物料需要补货。

## 5.4 供应商管理功能实现

供应商管理功能用于维护供应商的基本信息。

供应商管理页面的布局和物料管理类似，也是使用表格展示供应商列表。表格包含供应商编码、供应商名称、联系人、联系电话、地址等列。

新增和编辑供应商的功能实现方式和物料管理基本一致，都是通过对话框进行操作。供应商编码同样需要保证唯一性。

在实现过程中，供应商信息和物料信息是通过supplier_id字段关联的。在物料管理页面新增物料时，可以选择供应商。不过考虑到时间限制，这个关联功能做得比较简单，只是保存了供应商ID，没有在物料列表中显示供应商名称。

## 5.6 入库管理功能实现

入库管理是系统的核心业务功能之一，处理物料的入库操作。

### 5.6.1 入库记录列表

入库管理页面展示了所有的入库记录。表格包含入库单号、物料ID、仓位ID、数量、操作人、入库时间等列。

入库单号是系统自动生成的，格式为"IN"加上当前时间戳。这样可以保证入库单号的唯一性，而且从单号就能大概看出入库的时间。

表格支持分页显示，每页显示10条记录。如果入库记录很多，可以通过翻页查看。

[此处插入入库管理界面截图]

图5-6 入库管理界面

### 5.6.2 入库操作实现

点击"新增入库"按钮，会弹出入库信息录入对话框。对话框中需要选择物料、选择仓位、输入入库数量、输入操作人、填写备注。

物料和仓位都是通过下拉框选择的。下拉框的选项是从后端获取的，物料下拉框显示所有物料的名称，仓位下拉框显示所有仓位的名称。操作人默认填充当前登录用户的用户名。

点击保存后，前端会调用后端的入库接口。后端InboundService的save方法实现了入库的业务逻辑。这个方法使用了@Transactional注解，保证入库操作的事务性。

入库操作包含两个步骤：保存入库记录和更新物料库存。保存入库记录时，会自动生成入库单号和入库时间。更新物料库存时，会根据物料ID查询物料信息，然后将物料的当前库存加上入库数量，最后更新到数据库。

如果这两个步骤中任何一个失败，整个事务都会回滚，保证数据的一致性。比如入库记录保存成功了，但更新库存时出错，那么入库记录也会被回滚，不会保存到数据库。

### 5.6.3 库存自动更新

入库操作最重要的一点就是要同步更新物料的库存。在实现时，入库记录和库存更新是在同一个事务中完成的。

更新库存的逻辑很简单，就是查询物料的当前库存，加上入库数量，然后更新回数据库。使用MyBatis Plus的updateById方法可以很方便地完成这个操作。

在测试过程中，特意测试了事务回滚的情况。比如故意让更新库存的操作失败，发现入库记录确实没有保存，说明事务机制是有效的。

## 5.7 出库管理功能实现

出库管理的实现和入库管理类似，但多了一个库存检查的步骤。

### 5.7.1 出库操作实现

出库操作的界面和入库基本一致，也是通过对话框录入出库信息。需要选择物料、选择仓位、输入出库数量、输入操作人、填写备注。

出库单号的格式是"OUT"加上时间戳，和入库单号的生成方式类似。

后端OutboundService的save方法实现了出库的业务逻辑。出库操作比入库多了一个步骤，就是在保存出库记录之前，需要先检查库存是否充足。

库存检查的逻辑是：根据物料ID查询物料信息，判断物料的当前库存是否大于等于出库数量。如果库存不足，直接抛出异常，提示"库存不足"。前端收到这个异常后，会显示错误提示信息。

如果库存充足，就保存出库记录，然后更新物料库存。更新库存时，是将物料的当前库存减去出库数量。整个操作同样使用了事务管理，保证数据一致性。

### 5.7.2 库存不足处理

库存不足是出库操作中需要特别处理的情况。在实现时，如果检测到库存不足，后端会抛出RuntimeException异常。

前端在调用出库接口时，使用了try-catch来捕获异常。如果出库失败，会使用Naive UI的message组件显示错误提示。用户看到提示后，可以检查库存情况，或者减少出库数量。

这种处理方式比较简单直接，符合本科毕业设计的要求。在实际的生产系统中，可能需要更复杂的库存管理机制，比如预留库存、安全库存等。

## 5.8 库存查询功能实现

库存查询功能用于查看所有物料的当前库存情况。

库存查询页面展示了物料列表，包括物料编码、物料名称、分类、单位、当前库存、预警库存等信息。这个页面和物料管理页面看起来很像，但侧重点不同。物料管理侧重于维护物料的基本信息，库存查询侧重于查看库存状态。

对于库存低于预警值的物料，在表格中会用红色标注，提醒管理员注意。这个功能是通过自定义表格列的渲染函数实现的，判断当前库存是否小于等于预警库存，如果是就用红色显示。

页面还提供了搜索功能，可以按物料名称或编码搜索。这样在物料很多的情况下，可以快速找到想要查看的物料。

## 5.9 系统布局实现

系统的整体布局使用了Naive UI的Layout组件。布局分为左侧边栏、顶部导航栏和主内容区三部分。

### 5.9.1 侧边栏菜单

左侧边栏显示系统的主菜单，包括数据看板、物料管理、供应商管理、仓位管理、入库管理、出库管理、库存查询七个菜单项。

每个菜单项都配有图标，图标使用了ionicons5图标库。菜单项的激活状态会根据当前路由自动更新，当前页面对应的菜单项会高亮显示。

侧边栏顶部是系统的logo区域，显示"仓库管理系统"的标题。这个区域使用了渐变色背景，和登录页面的主题色保持一致。

### 5.9.2 顶部导航栏

顶部导航栏左侧显示当前页面的标题，右侧显示当前登录用户的用户名。

用户名区域是一个下拉菜单，点击后可以选择退出登录。退出登录时，会清除localStorage中保存的用户信息和token，然后跳转到登录页面。

### 5.9.3 主内容区

主内容区使用router-view组件，根据当前路由显示对应的页面组件。内容区有24px的内边距，让页面内容不会紧贴边缘。

整个布局的高度是100vh，占满整个视口高度。侧边栏的宽度固定为240px，内容区自适应剩余宽度。

## 5.10 前后端数据交互

系统采用前后端分离架构，前后端通过HTTP接口进行数据交互。

### 5.10.1 统一返回格式

后端所有接口都使用统一的返回格式。定义了一个Result类，包含code、message、data三个字段。code表示状态码，200表示成功，500表示失败。message是提示信息，data是具体的数据。

这样前端在处理响应时就很方便，只需要判断code是否为200，就知道请求是否成功。如果成功，从data字段获取数据；如果失败，从message字段获取错误信息。

### 5.10.2 Axios请求封装

前端使用Axios库发送HTTP请求。为了方便使用，对Axios进行了封装，创建了一个request实例。

在请求拦截器中，会自动在请求头中添加token。token从localStorage中获取，这样每次请求都会携带token，后端可以通过token验证用户身份。

在响应拦截器中，会自动提取响应数据中的data字段。这样在调用接口时，直接就能拿到业务数据，不需要再手动提取。

如果请求失败，响应拦截器会捕获错误，在控制台输出错误信息。这样在开发调试时可以方便地看到错误原因。

### 5.10.3 API接口管理

前端将所有的API接口调用封装在api目录下的文件中。每个模块对应一个文件，比如user.js封装用户相关的接口，material.js封装物料相关的接口。

每个接口函数都返回一个Promise，调用时使用async/await语法。这样代码看起来更简洁，也更容易处理异步操作。

举个例子，登录接口的调用：

```javascript
export const login = (data) => {
  return request.post('/api/user/login', data)
}
```

在组件中使用：

```javascript
const res = await login(formData.value)
```

这种方式让接口调用变得很简单，而且所有接口都集中管理，方便维护。

## 5.11 状态管理实现

系统使用Pinia进行状态管理，主要管理用户的登录状态。

创建了一个userStore，包含user和token两个状态。user存储用户信息，token存储登录凭证。

userStore提供了setUser、setToken、logout三个方法。setUser和setToken用于设置用户信息和token，logout用于清除用户状态。

在登录成功后，会调用setUser和setToken保存用户状态。在其他页面需要获取用户信息时，可以通过userStore.user访问。退出登录时，调用logout清除状态。

状态管理让用户信息在整个应用中共享变得很方便，不需要在每个组件中都从localStorage读取。

## 5.12 路由配置实现

系统使用Vue Router进行路由管理。路由配置分为两部分：登录路由和系统路由。

登录路由比较简单，路径是/login，对应Login组件。

系统路由使用了嵌套路由，父路由对应Layout组件，子路由对应各个功能页面。这样所有的功能页面都会显示在Layout的内容区中，共享侧边栏和顶部导航栏。

路由配置中设置了路由守卫，在beforeEach钩子中检查用户是否登录。如果访问系统页面但没有登录，会跳转到登录页。如果已经登录但访问登录页，会跳转到首页。

路由使用了懒加载方式，通过import()函数动态导入组件。这样可以实现按需加载，减少首次加载的资源大小，提高页面加载速度。

## 本章小结

这一章详细介绍了智能仓库管理系统各个功能模块的实现过程。从用户登录开始，到系统首页的数据统计和图表展示，再到物料管理、供应商管理、仓位管理等基础功能，以及入库出库等核心业务功能，最后是库存查询和系统布局的实现。在实现过程中，前端使用Vue 3和Naive UI构建用户界面，后端使用Spring Boot和MyBatis Plus处理业务逻辑，前后端通过RESTful API进行数据交互。整个系统的功能基本实现了预期的目标，能够满足仓库管理的基本需求。
